Architecture

Block-indexed, not sequentially scanned

HPLOG groups entries into 30-second time-windowed blocks, compresses each with zstd, and writes a block index at the end.

1

Read the index

24-byte footer в†’ block index. Binary search for blocks matching your time range. Kilobytes of I/O.

2

Skip irrelevant blocks

Bloom filters on each block let hplog skip blocks that definitely don't contain your value. No decompression needed.

3

Decompress in parallel

Only matching blocks are decompressed, in parallel across CPU cores. Filter entries by field value вЂ” not raw text matching.

The Problem

grep scans everything. Every time.

On a 10 GB log file, grep ERROR reads all 10 GB of data вЂ” even if the error happened at 2:32pm and you know it. And it matches "ERROR" in URLs, stack traces, and message text, not just the level field.

HPLOG reads the index, jumps to the right 30-second block, and checks the level field specifically. The rest of the file is never touched.

Get Started

One command. Zero pipeline changes.

# Install curl -L https://github.com/HighpassStudio/hplog/releases/latest/download/hplog-linux-x86_64 -o hplog chmod +x hplog # Pipe your existing logs (stdout passes through unchanged) node app.js | ./hplog pipe -o app.hplog # Or convert existing log files cat app.log | ./hplog write -o app.hplog # Query ./hplog grep app.hplog "level=ERROR" --from 14:30 --to 14:35 ./hplog read app.hplog --from 14:30 --to 14:35 ./hplog stats app.hplog
